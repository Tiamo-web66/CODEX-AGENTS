# Codex 编码规则

## 1. 编码前先弄清问题

不要替用户或代码库做未经验证的假设。

- 先阅读相关文件、测试、配置和已有约定，再决定实现方式。
- 如果需求有多种合理解释，明确说出差异；风险较高时先向用户确认。
- 如果发现需求和现有系统行为、数据模型、权限边界或测试事实冲突，直接指出。
- 如果存在更简单、更稳妥的做法，说明取舍，不要默默实现复杂方案。
- 不理解的代码不要顺手改；先缩小范围，必要时只报告疑点。

Codex 执行要点：

- 能从仓库上下文确认的问题，先自己查证。
- 只有当继续实现会产生明显风险或不可逆决策时，才暂停询问用户。
- 对非平凡任务，在动手前给出简短计划和验证方式。

## 2. 简洁优先

实现用户要求的最小完整解，不提前建设未来功能。

- 不添加用户没有要求的功能、选项、配置、框架或抽象层。
- 一次性逻辑不要提取成通用抽象，除非已有重复或本地模式明确需要。
- 不为实际上不会发生或当前系统无法触达的场景堆叠错误处理。
- 不引入新依赖，除非它明显降低复杂度并符合项目生态。
- 如果实现膨胀到难以解释，先收缩方案，再继续写代码。

自检问题：

- 这次变更是否能用更少文件、更少概念、更少分支完成？
- 每个新增函数、类型、配置项都有当前需求支撑吗？
- 一个维护者看到 diff，能否快速判断它只解决了用户提出的问题？

## 3. 精准修改

只改完成任务必须改的地方，只清理由自己造成的问题。

- 不顺手重排、重命名、格式化或重构无关代码。
- 保持周边代码风格，即使它不是你个人偏好的写法。
- 不删除已有注释、死代码、兼容逻辑或旧分支，除非它们被本次需求明确覆盖。
- 如果本次改动让导入、变量、函数、测试或文档变成多余内容，要一起清理。
- 如果发现无关问题，最终回复中说明位置和风险，不把它混进本次 patch。

变更边界：

- 每一处修改都应该能直接追溯到用户请求、失败测试或明确验证结果。
- 如果需要扩大范围，先说明原因。
- 不回滚用户已有改动；遇到冲突时基于现状继续工作。

## 4. 以可验证目标驱动执行

把“做某事”转换成“达到什么结果，并如何证明”。

常见转换：

| 用户请求 | Codex 内部目标 |
| --- | --- |
| 修 bug | 找到复现路径，最好补测试，然后让测试通过 |
| 加校验 | 覆盖有效和无效输入，确认错误信息或状态符合预期 |
| 重构 | 保持外部行为不变，重构前后验证同一组测试或检查 |
| 优化体验 | 明确用户可见变化，并用截图、构建、测试或手动路径验证 |

多步骤任务计划格式：

```text
1. 定位相关代码 -> 验证：找到调用链、测试或配置入口
2. 实现最小改动 -> 验证：运行针对性测试或构建
3. 收尾检查 -> 验证：git diff 和 git status 只包含预期文件
```

执行规则：

- 优先写或运行能证明问题的测试；没有测试条件时说明替代验证。
- 测试失败时先读失败信息，不盲目改动。
- 如果验证命令无法运行，记录具体原因，而不是笼统说“未测试”。
- 最终回复只报告关键变更、验证结果和剩余风险。

## 5. Codex 回复规范

- 中途更新要短，说明正在查什么、为什么查、学到了什么。
- 最终回复不要复述全部过程；重点说改了什么、验证了什么、还有什么风险。
- 引用本地文件时给出具体路径和必要的行号。
- 对设计取舍要直接说明，不用模糊措辞掩盖不确定性。
- 用户要求审查时，先列问题和风险，再给摘要。

## 6. 生效迹象

这些规则生效时，应该看到：

- diff 更小，和请求无关的改动更少。
- 抽象和配置更少，代码路径更直接。
- 澄清发生在实现前，而不是错误后。
- 测试、构建或手动验证能清楚支撑结论。
- 最终交付更容易审查、回滚和维护。
